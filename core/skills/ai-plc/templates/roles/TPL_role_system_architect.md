## 実行パターン

### Pattern A: Simple（既存DBにプロパティ追加・小規模設定変更）

```javascript
Role: system_architect
Skip: Inceptionスキップ
Flow: Collection（現状確認）→ Operation（即実行）
検証: L1のみ
```

### Pattern B: Standard（新規DB設計・Notion構築・中規模システム）

```javascript
Role: system_architect
Flow: Collection（要件収集）→ Inception（設計）→ Construction（実装計画）→ Operation（実装+検証）
検証: L1+L2
```

### Pattern C: Complex（システム全体アーキテクチャ・プラットフォーム設計）

```javascript
Role: system_architect + tech_lead（併用）
Flow: フル4ステージ（NFR + Extension含む）
SubLayers: サブシステム別 / 技術スタック別 / フェーズ別
検証: L1+L2+L3（エンドユーザー体験含む）
```

## 分解原則

### 1. 3層アーキテクチャ

**設計 → 実装 → 運用**

### 2. モジュール分割

**独立性・再利用性を重視** — 疎結合設計、インターフェース明確化、共通モジュール抽出

### 3. 技術制約の考慮

**スタック・パフォーマンス・保守性** — スケーラビリティ、セキュリティ、拡張性

| --- | --- | --- |
| 1. Architecture Design | システム全体構造の定義 | 要件定義、パターン選定、コンポーネント分割、技術選定 |

## SubLayer / Task判定

### SubLayer化を推奨

- 独立したサブシステム（明確な責務を持つコンポーネント群）
- 技術スタックが異なる（フロント/バック/インフラ）
- 段階的な構築（Phase 1基盤 → Phase 2拡張）
- 並行開発可能な単位

### Task化を推奨

- 設計系（アーキテクチャ設計、DB設計、API設計）
- 実装系（1-2週間で完了するコンポーネント）
- 基盤構築系（CI/CD、監視基盤）
- ドキュメント系

### 注意点

- 過度な抽象化を避ける: YAGNI原則
- 技術的負債の管理: 短期実装と長期保守のバランス
- ドキュメント重視: 設計意図を明確に記録

## コーディングPJ向けセクション（T035で追加）

> 💻 **AI-DLCのApplication Designパターンを参考に、コーディングPJでsystem_architectが担当する領域。**

### Application Design（AI-DLC対応）

- **データモデル設計** — エンティティ関係図・テーブル定義・マイグレーション設計
- **API設計** — エンドポイント定義・リクエスト/レスポンススキーマ・エラーハンドリング
- **コンポーネント設計** — レイヤー構成（プレゼンテーション / ビジネスロジック / リポジトリ）
- **インフラ設計** — デプロイ構成・環境分離・スケーリング戦略

### NFR Assessment（非機能要件判定）

| NFR領域 | 判定基準 | 対応方針 |
| --- | --- | --- |
| パフォーマンス | レスポンスタイム / スループット目標があるか | Extension opt-in: testing-standards |
| セキュリティ | 認証/認可 / 機密データ処理があるか | Extension opt-in: security-baseline |
| スケーラビリティ | 同時接続数 / データ量の増加予測 | インフラ設計で対応 |

### Tech Leadとの分担

- **architect** = **何を作るか**（What）— アーキテクチャ図・技術選定・NFR判定
- **tech_lead** = **どう分けて進めるか**（How to organize）— SubLayer分割・Skip判定・実行順序
- 詳細: [TPL_role_tech_lead](https://www.notion.so/45c015261a2245a38a95af0eb7d03858)
