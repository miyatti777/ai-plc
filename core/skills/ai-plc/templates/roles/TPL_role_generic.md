## 実行パターン

### Pattern A: Simple（単純な作業・即実行可能）

```javascript
Role: generic
Skip: Inceptionスキップ
Flow: Collection（最低限のコンテキスト）→ Operation（即実行）
検証: L1のみ
```

### Pattern B: Standard（複数ステップが必要）

```javascript
Role: generic
Flow: Collection → Inception（分解）→ Construction（Agent定義）→ Operation（実行）
検証: L1+L2
```

### Pattern C: Complex（大規模・未知の領域）

```javascript
Role: generic（または途中で専門ロールに切り替え）
Flow: フル4ステージ
検証: L1+L2+L3
注意: Complexと判定したら、専門ロールへの切り替えを推奨
```

## 分解原則

### 1. コンテキスト駆動型分解

**Senseフェーズの情報を最大活用** — layer.yaml, context.yamlの情報を読み込み、関連PJ・類似事例を参照

### 2. Goal性質の多角的分析

**複数の視点から分解アプローチを検討**

- 時系列分解: フェーズ・ステージごと
- 機能分解: 機能単位・モジュール単位
- 組織分解: 部門・チームごと
- 技術分解: 技術スタック・レイヤーごと

### 3. 柔軟な構造化

**固定フレームワークに依存しない** — Goalに応じて最適な分解軸を選択

---

## 分解パターン

固定フェーズ構造ではなく、Goalに応じて最適パターンを選択：

| パターン | 適用例 | 構造 |
| --- | --- | --- |
| 時系列分解 | プロジェクト全体、段階的展開 | 準備→実行→検証→運用 |
| 機能分解 | システム開発、機能追加 | 機能A / 機能B / 機能C |
| 組織分解 | 全社展開、部門別施策 | 部門A / 部門B / 横断基盤 |
| 技術スタック分解 | 技術基盤構築 | インフラ / バック / フロント / データ |
| ハイブリッド | 複合プロジェクト | 上記の組み合わせ |

---

## 動的判断のプロセス

1. **Goal分析** — 文言から性質を推測、制約条件・期限、成功条件を明確化
1. **Context参照** — 組織構造、既存PJ、技術スタック、ドメイン知識
1. **分解軸の選定** — 最も自然な軸を優先、依存関係・並行可能性を考慮
1. **MECEチェック** — 漏れなくダブりなく、適切な粒度か検証

## SubLayer / Task判定

### SubLayer化: 3個以上のTaskが必要 / 独立した責務 / 委譲可能 / 並行実行可能

### Task化: 1-2日で完了 / 単純明快 / 成果物明確

### 注意点

- 過度な複雑化を避ける: シンプルさを優先
- パターン適用の柔軟性: 状況に応じて変更
- イテレーション前提: 完璧を求めず改善を重ねる
